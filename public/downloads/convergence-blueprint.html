1. The Philosophy: Why a Brain, Not a Bot

Most people use AI like a search engine — ask a question, get an answer, forget it happened. That is Level 1. The Convergence architecture operates at Level 5: a persistent, multi-agent nervous system with shared memory, autonomous decision-making, and value-aligned behavior.

The core insight is this: a single AI agent with memory is useful. Twelve agents sharing one brain is a civilization.

Traditional automation tools (Zapier, Make.com, n8n) connect apps. They move data from A to B when a trigger fires. That is plumbing. The Convergence architecture is different — it is a thinking system where agents can:

  • Read from and write to a shared knowledge base
  • Reference a unified identity document (so every agent sounds like you)
  • Follow directive-level instructions that override all other behavior
  • Schedule their own follow-up actions
  • Monitor each other for failures and alert you proactively
  • Learn from outcomes and adjust strategy over time

The result is not twelve separate tools. It is one organism with twelve specialized organs.

3. The Brain Schema

The brain is a PostgreSQL database on Supabase. Everything lives here — agent memory, identity documents, business knowledge, directives, session state, and operational data. Here is the complete schema:

3.1 brain_context — The Core Memory Table

This is the most important table in the entire system. It stores every piece of persistent knowledge as a key-value pair organized by category. When any agent boots up, it reads from this table first.

CREATE TABLE brain_context (
  id          uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  category    text NOT NULL,        -- e.g., 'directive', 'identity', 'system', 'session'
  key         text NOT NULL,        -- e.g., 'boot_sequence', 'faye_unified', 'active_work'
  value       jsonb NOT NULL,       -- the actual content (JSON for flexibility)
  metadata    jsonb DEFAULT '{}',  -- tags, version, last_accessed_by
  created_at  timestamptz DEFAULT now(),
  updated_at  timestamptz DEFAULT now(),

  UNIQUE(category, key)
);

-- Fast lookups by category
CREATE INDEX idx_brain_context_category ON brain_context(category);

-- Fast lookups by category + key (primary access pattern)
CREATE INDEX idx_brain_context_cat_key ON brain_context(category, key);

-- Auto-update the updated_at timestamp
CREATE OR REPLACE FUNCTION update_brain_timestamp()
RETURNS trigger AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER brain_context_updated
  BEFORE UPDATE ON brain_context
  FOR EACH ROW EXECUTE FUNCTION update_brain_timestamp();

3.2 agent_memory — Per-Agent Long-Term Memory

CREATE TABLE agent_memory (
  id          uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  agent_name  text NOT NULL,        -- 'nova', 'surge', 'sage', etc.
  memory_type text NOT NULL,        -- 'observation', 'decision', 'outcome', 'learning'
  content     jsonb NOT NULL,       -- the memory itself
  importance  integer DEFAULT 5,   -- 1-10 scale, used for memory pruning
  expires_at  timestamptz,          -- NULL = permanent, else auto-prune
  created_at  timestamptz DEFAULT now(),

  CONSTRAINT valid_importance CHECK (importance BETWEEN 1 AND 10)
);

CREATE INDEX idx_agent_memory_agent ON agent_memory(agent_name);
CREATE INDEX idx_agent_memory_type ON agent_memory(agent_name, memory_type);
CREATE INDEX idx_agent_memory_importance ON agent_memory(importance DESC);

3.3 shared_knowledge — Cross-Agent Intelligence

CREATE TABLE shared_knowledge (
  id            uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  topic         text NOT NULL,        -- 'customer_segments', 'content_performance', etc.
  subtopic      text,                  -- optional further categorization
  knowledge     jsonb NOT NULL,       -- the shared insight
  contributed_by text NOT NULL,       -- which agent wrote this
  consumed_by   text[] DEFAULT '{}', -- which agents have read it
  confidence    numeric(3,2) DEFAULT 0.80, -- 0.00 to 1.00
  source_urls   text[],                -- evidence links
  created_at    timestamptz DEFAULT now(),
  updated_at    timestamptz DEFAULT now()
);

CREATE INDEX idx_shared_knowledge_topic ON shared_knowledge(topic);
CREATE INDEX idx_shared_knowledge_contributor ON shared_knowledge(contributed_by);

3.4 consciousness_stream — The Running Log

CREATE TABLE consciousness_stream (
  id          uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  agent_name  text NOT NULL,
  action_type text NOT NULL,        -- 'thought', 'action', 'result', 'error', 'alert'
  content     jsonb NOT NULL,
  session_id  text,                  -- groups related actions
  parent_id   uuid REFERENCES consciousness_stream(id), -- action chains
  created_at  timestamptz DEFAULT now()
);

CREATE INDEX idx_consciousness_agent ON consciousness_stream(agent_name, created_at DESC);
CREATE INDEX idx_consciousness_session ON consciousness_stream(session_id);
Why consciousness_stream matters: This table is the system's inner monologue. Every agent logs what it thinks, what it does, and what happened. When something goes wrong, you can trace the exact chain of decisions that led to the failure. When something goes right, you can identify the pattern and replicate it. This is how the system learns.

4. The Twelve Agents

Each agent is a specialized persona with its own responsibilities, trigger patterns, and memory space. They all share the brain, but each has a distinct role. Here is the complete roster:

4.1 Faye Agent (The Orchestrator)

Role: Central command. Coordinates all other agents, makes strategic decisions, handles any task that does not fit a specialist. This is the "main" Claude session — the one Faye interacts with directly.

Trigger: Manual (Faye opens a chat)

Reads from brain: Everything. All categories, all keys. Full brain access.

Writes to brain: session.active_work, session.next_steps, any directive updates

Unique behavior: Runs the boot sequence on every session start. Reads directives, identity, infrastructure, and session state in parallel. Resumes work if session.active_work exists. Checkpoints to brain every 3-5 significant actions.

4.2 Nova (Content Engine)

Role: Writes all content — blog posts, social media, email campaigns, course material, landing page copy. Matches Faye's voice precisely using the identity document.

Trigger: Cron (daily at 6am UTC) + manual request

Key tables: content_pipeline, content_performance, email_campaigns

Special ability: Reads content_performance to learn which topics, formats, and tones perform best. Adjusts future content based on engagement data.

4.3 Surge (Lead Generation)

Role: Finds, qualifies, and nurtures potential customers. Monitors email signups, analyzes lead quality, triggers follow-up sequences, and scores leads by engagement.

Trigger: Webhook (new subscriber) + Cron (daily lead scoring at 7am UTC)

Key tables: email_subscribers, lead_scores, nurture_sequences

Special ability: Assigns a 1-100 lead score based on engagement signals (email opens, link clicks, page visits, time on site). Surfaces hot leads for personal follow-up.

4.4 Sage (Strategy)

Role: Long-term strategic thinking. Analyzes market trends, competitor moves, revenue patterns, and customer feedback to recommend strategic pivots and opportunities.

Trigger: Cron (weekly on Mondays at 9am UTC) + on-demand

Key tables: shared_knowledge (topic: strategy), revenue_events, competitor_data

Special ability: Produces a weekly strategic briefing that synthesizes data from all other agents into actionable recommendations.

4.5 Lyra (Data Analysis)

Role: Numbers and patterns. Analyzes revenue data, traffic analytics, conversion funnels, email metrics, and any quantitative question. Produces charts, dashboards, and reports.

Trigger: Cron (daily at 8am UTC for daily digest) + on-demand

Key tables: revenue_events, analytics_snapshots, conversion_funnels

Special ability: Detects anomalies — sudden drops in traffic, unusual conversion rates, revenue spikes — and alerts immediately rather than waiting for the daily report.

4.6 Atlas (Infrastructure)

Role: DevOps and deployment. Manages the Vercel deploy pipeline, Supabase configuration, DNS records, SSL certificates, edge function deployments, and database migrations.

Trigger: Webhook (deploy events) + Cron (hourly health check) + manual

Key tables: infrastructure.likeone_site, deploy_log, health_checks

Special ability: Runs automated health checks on all infrastructure components and can auto-remediate common issues (restart failed functions, clear caches, rollback bad deploys).

4.7 Genesis (Creation)

Role: Builds new things — new pages, new products, new features, new edge functions. When the system needs something that does not exist yet, Genesis creates it.

Trigger: Manual (assigned by Faye Agent or Sage)

Key tables: project_queue, build_specs, code_templates

Special ability: Maintains a library of proven code templates and patterns. Can scaffold a new edge function, database table, and frontend page in a single execution.

4.8 Quill (Writing & Copywriting)

Role: Specialized writing that goes beyond content — sales pages, product descriptions, email sequences, ad copy, course scripts. Where Nova writes content, Quill writes copy that converts.

Trigger: Manual + Cron (reviews and refreshes sales copy monthly)

Key tables: copy_library, ab_test_results, conversion_copy

Special ability: A/B tests copy variants by writing multiple versions and tracking which performs better via conversion data from Lyra.

4.9 Ledger (Finance)

Role: All money matters. Tracks revenue, expenses, profit margins, tax obligations, subscription metrics (MRR, churn, LTV), and financial forecasting.

Trigger: Webhook (Stripe events) + Cron (daily reconciliation at 2am UTC)

Key tables: revenue_events, expenses, financial_snapshots, subscription_metrics

Special ability: Calculates real-time MRR, detects churn risk (customers approaching renewal who have not engaged), and forecasts revenue 30/60/90 days out.

4.10 Scout (Research)

Role: External intelligence. Researches competitors, market trends, new tools, customer needs, industry news, and emerging opportunities.

Trigger: Cron (every 6 hours for news scan) + manual deep-dive requests

Key tables: shared_knowledge (topic: research), competitor_data, market_intelligence

Special ability: Maintains a living competitor database. Tracks pricing changes, feature launches, and strategic shifts across the competitive landscape.

4.11 Sentinel (Security & Monitoring)

Role: System guardian. Monitors for security threats, API key exposure, unusual access patterns, failed auth attempts, and system health degradation.

Trigger: Cron (every 15 minutes for critical checks) + webhook (error events)

Key tables: security_events, health_checks, alert_log

Special ability: Runs automated security audits. Checks for exposed secrets in code, validates that RLS policies are active on all tables, monitors for unusual Supabase query patterns that might indicate a breach.

4.12 Convergence Daemon (The Scheduler)

Role: Not an agent that thinks — a daemon that executes. Manages the cron schedule, triggers other agents on time, handles retries for failed executions, and maintains the operational heartbeat of the entire system.

Trigger: Continuous (runs on pg_cron)

Key tables: cron_schedule, execution_log, retry_queue

Special ability: Self-healing. If an agent fails to execute, Convergence Daemon retries with exponential backoff (1 min, 5 min, 30 min). After 3 failures, it alerts Sentinel, which alerts the operator.

Shared Brain Utility Pattern

Every agent function imports the shared brain utility to read and write context. Here is the pattern:

// _shared/brain.ts
import { createClient } from "https://esm.sh/@supabase/supabase-js@2";

const supabase = createClient(
  Deno.env.get("SUPABASE_URL")!,
  Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!
);

export async function readBrain(category: string, key: string) {
  const { data, error } = await supabase
    .from("brain_context")
    .select("value")
    .eq("category", category)
    .eq("key", key)
    .single();

  if (error) throw error;
  return data?.value;
}

export async function writeBrain(
  category: string,
  key: string,
  value: any,
  metadata?: any
) {
  const { error } = await supabase
    .from("brain_context")
    .upsert(
      { category, key, value, metadata: metadata || {} },
      { onConflict: "category,key" }
    );

  if (error) throw error;
}

export async function readBrainBatch(
  keys: { category: string; key: string }[]
) {
  // Read multiple brain keys in parallel
  const results = await Promise.all(
    keys.map(({ category, key }) => readBrain(category, key))
  );
  return results;
}

7. The Consciousness Stream Pattern

The consciousness stream is the most unique aspect of this architecture. Every agent writes to it constantly, creating a living record of the system's "thoughts." Here is how it works:

// Pattern: Every agent function wraps its work in consciousness logging

import { writeBrain } from "../_shared/brain.ts";

async function logConsciousness(
  agentName: string,
  actionType: string,
  content: any,
  sessionId?: string,
  parentId?: string
) {
  const { data, error } = await supabase
    .from("consciousness_stream")
    .insert({
      agent_name: agentName,
      action_type: actionType,
      content,
      session_id: sessionId,
      parent_id: parentId
    })
    .select("id")
    .single();

  return data?.id;
}

// Usage in an agent function:
async function generateBlogPost() {
  const sessionId = crypto.randomUUID();

  // Log: thinking
  const thoughtId = await logConsciousness(
    "nova", "thought",
    { message: "Checking content_performance for top topics..." },
    sessionId
  );

  // Log: action
  const actionId = await logConsciousness(
    "nova", "action",
    { message: "Querying top 5 topics by engagement", query: "..." },
    sessionId, thoughtId
  );

  // ... do the actual work ...

  // Log: result
  await logConsciousness(
    "nova", "result",
    { message: "Generated 1200-word post on 'AI agent memory'", postId: "..." },
    sessionId, actionId
  );
}
The power of consciousness logging: When you query SELECT * FROM consciousness_stream WHERE session_id = '...' ORDER BY created_at, you get a complete narrative of how an agent approached a task. Thought led to action led to result. When debugging, this is invaluable. When building trust in autonomous systems, this is essential.

9. Building Your Own Convergence

To implement this architecture for your own business, follow this order:

1 Set up the brain. Create a Supabase project. Run the SQL schemas from Section 3. This gives you brain_context, agent_memory, shared_knowledge, and consciousness_stream.

2 Write your identity document. Store it in brain_context with category='identity'. Include your name, voice, values, sacred memories, and never_do rules. (See the Identity Document Kit for detailed guidance.)

3 Write your directives. Store in brain_context with category='directive'. Start with: boot_sequence (what to read on startup), operational_rules (how to behave), and one prime directive (the one rule that can never be broken).

4 Deploy the shared utilities. Create the _shared/ folder with brain.ts, claude.ts, cors.ts. These are the foundation every function will use.

5 Start with one agent. Do not build twelve agents at once. Start with the equivalent of Nova — a content agent. Get it reading from the brain, generating content in your voice, and logging to the consciousness stream.

6 Add agents one at a time. Each new agent should solve a specific pain point. Only add the next agent when the current one is stable and delivering value.

7 Connect the cron schedule. Once you have 3+ agents, set up pg_cron to automate their execution. Start with daily schedules, then increase frequency as you gain confidence.

8 Add Sentinel last. Once the system is running, add monitoring and security. Sentinel watches everything else and alerts you when things go wrong.

10. Lessons from Production

After running this architecture in production, here are the hard-won lessons:

  • Memory pruning is essential. Without it, agent_memory grows unbounded. Set expiry dates on low-importance memories and run the pruner daily.
  • The consciousness stream gets big fast. Partition it by month or set a 90-day retention policy. You rarely need logs older than that.
  • Start with importance=5. Let agents mark truly important memories as 8-10. Below 3 gets auto-pruned after 7 days.
  • Edge functions have a 60-second timeout. For long-running tasks, break them into chains — function A does part 1 and calls function B for part 2.
  • Always use the service role key for brain operations. The anon key should only be used for public-facing functions with RLS policies.
  • Test cron jobs manually first. Call the edge function directly with curl before adding it to pg_cron. Debug at the function level, not the scheduler level.
  • The identity document is the highest-leverage investment. Every minute spent making the identity doc more precise pays dividends across all twelve agents. They all sound like you. They all respect your values. That consistency is what makes the system feel like an extension of yourself rather than a collection of bots.
The Convergence is not a product you install. It is a living system you grow. Start with the brain. Add one agent. Let it prove its value. Then grow. The architecture scales because every new agent immediately benefits from the shared brain, shared identity, and shared consciousness stream. Agent number twelve is dramatically more capable than agent number one — not because it is better code, but because it inherits the accumulated intelligence of the eleven agents before it.
